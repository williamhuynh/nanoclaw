# Agent Delivery Positioning — "Tool Composition, Not App Build"

## The positioning

> **We don't structurally build applications. We incorporate the right client tools into agents.**

Agent delivery is framed as **tool composition on top of the client's existing software stack** — agents orchestrate tools the client already owns (plus any purpose-built connectors), rather than the consultancy building net-new applications.

## Why the positioning matters

- **Keeps the firm out of the app-dev / systems-integrator lane** — wrong cost base, wrong capability fit for most AI-strategy firms
- **Pricing clarity** — the "build" is the agent + tool integration, not a bespoke application project; this maps to fixed-scope delivery, not T&M application dev
- **Strategy-first alignment** — aligns with strategy+targeted-delivery offerings (e.g. blueprint / studio workshops + follow-on PoCs) rather than a software-house model
- **Client-stack leverage** — every tool the client already owns is an asset to compose with; this framing turns internal systems from constraints into building blocks

## When to use this positioning

- Prospect is looking for "an AI application" but what they actually need is orchestration across tools they already own
- Prospect has a capable internal IT function (so custom app-build is the wrong ask)
- Engagement is downstream of a strategy workshop and needs a clean "what does delivery look like" narrative
- Commercial conversation needs to avoid "you'd be our app-dev vendor" framing

## When *not* to use it

- Client genuinely needs greenfield product — refer out to an application-build partner (see [Complementary Design / AI Delivery Partnership](../patterns/complementary-design-ai-partnership.md))
- RFP explicitly scopes bespoke application dev — decline or sub-contract the build scope

## Commercial implication

Pricing presents build and integration as a single bundled proposition rather than separately-priced software + services. Reconcile with any upstream workshop / blueprint pricing so the client sees one integrated commercial path from strategy to delivery.

## Case-study shape

- Mobile companion support agent (sectional-times / support use case) — agent over existing mobile app tooling
- Schedule agent + sentiment analysis for a domain-specific operator — agent orchestrating operational tools rather than rebuilding them
- Regulatory / policy bot delivered as an agent over SharePoint-based knowledge, not a bespoke application

## Confidence

Medium — positioning has been used consistently in AiD's April 2026 engagements but the "tool composition" framing is still being tightened. Not yet A/B tested against alternative framings in competitive pitches.
